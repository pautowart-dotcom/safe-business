# Пакет 4 — Единая система сроков, тест и генерация журналов
*Для передачи Claude Code · подготовлено 22 июля 2026*

## Принципы этапа (прочитать перед началом работы)

1. **Тест безопасности** остаётся диагностикой — коротким инструментом "чего не хватает". Он НЕ собирает точные даты и не должен разрастаться в анкету на полчаса.
2. **"Мои сроки"** — новая вкладка внутри раздела "Безопасность" (не отдельный пункт нижней навигации), где пользователь по желанию вносит конкретные даты для календаря. Всё опционально: не заполнено поле — просто нет напоминания по нему, без блокировок и без "заполните обязательно".
3. **Единый источник дедлайнов.** Любой модуль (тест, сотрудники, документы, журналы, финансы), который порождает что-то со сроком, обязан регистрировать это в общей таблице `deadlines`, а не хранить дату у себя локально. Категория присваивается по смыслу (кадровые/помещение/юр.документы/налоги/журналы).
4. **Печать журналов бесплатна**, входит в подписку. Никакой логики оплаты/заказа за печать не нужно — только генерация файла.
5. **Не пытаться вычислить то, что нельзя вычислить.** Даты, зависящие от конкретных условий (кто выдал ЭЦП, условия договора аренды, срок на бирке огнетушителя) — только вводятся пользователем напрямую, никогда не угадываются системой. Вычислять можно только то, что фиксировано законом одинаково для всех (сроки налогов по режиму, СОУТ — 5 лет).

---

## Этап 1 — Расширение движка дедлайнов

Существующая таблица `deadlines` (категории legal/financial/staff) становится единственным местом хранения сроков.

Добавить категории (если ещё нет): `premises` (помещение/оборудование), `documents` (юр.документы), `journals` (журналы).

Ввести два типа сущностей в этой системе, выводящихся вместе:
- **Дедлайны** — есть точная дата (аренда, ЭЦП, налоги, мед.книжка).
- **Действия** — есть условие, но нет даты (не пройден тест, кончаются расходники, не заполнен журнал за сегодня).

---

## Этап 2 — Раздел "Безопасность": две вкладки

Внутри существующего раздела "Безопасность" — два таба:

### Таб 1: "Тест" (существующий, доработать)
Остаётся диагностикой. Убрать из него любые вопросы, которые сейчас пытаются вычислить точную дату (например, если сейчас спрашивается "когда получена ЭЦП" с попыткой прибавить срок — убрать саму попытку вычисления; сам факт "есть ли ЭЦП" может остаться в тесте как диагностический вопрос, но конкретная дата окончания переезжает в "Мои сроки").

**Важно:** проверить и переработать существующий механизм формирования PDF-отчёта по тесту — он не должен ссылаться на поля/даты, которые теперь убраны из теста. Отчёт остаётся содержательным и цельным без точных дат внутри (даты показываются отдельно, в "Мои сроки"/календаре).

### Таб 2: "Мои сроки" (новый)
Список категорий, каждая — карточка с полем "дата" (плюс опционально периодичность и заметка/контакт подрядчика). Все поля необязательны для заполнения.

**Полный список категорий, ничего не упускать:**

*Кадровые*
- Медицинские книжки сотрудников (у каждого сотрудника отдельно, дата окончания)
- Договоры с сотрудниками, если срочные (дата окончания)
- Повторный инструктаж по охране труда (дата следующего)
- СОУТ — спецоценка условий труда (дата последней; можно автоматически прибавлять 5 лет, это фиксировано законом)

*Помещение и оборудование*
- Договор аренды (дата окончания)
- Огнетушители (дата следующей перезарядки/поверки — с бирки самого огнетушителя)
- Техобслуживание пожарной сигнализации (дата следующего, периодичность по договору)
- Замер сопротивления изоляции электропроводки (дата следующей проверки)
- Договор на дезинфекцию/дератизацию помещения (дата окончания)
- УФ-лампы (уже реализовано — оставить как есть)

*Юридические документы*
- Электронная подпись, ЭЦП (дата окончания — спрашивать напрямую, срок зависит от способа выдачи и не вычисляется)
- Патент, если ИП на патентной системе (дата окончания — выбирается самим ИП при подаче, не вычисляется)
- Лицензии, если применимо к виду услуг (дата окончания)
- Договор на вывоз медицинских отходов класса Б (дата окончания)
- Договор на вывоз ТБО (дата окончания)

*Налоги и финансы (можно частично вычислять — фиксировано законом)*
- Фиксированные страховые взносы ИП (до конца года — вычисляется)
- 1% с дохода свыше 300 тыс. (до 1 июля следующего года — вычисляется)
- Налог по выбранному режиму (патент/УСН — сроки зависят от режима, но фиксированы законом для каждого режима — вычисляется, если известен режим)
- Отчётность за сотрудников (РСВ, 6-НДФЛ), если есть наёмные — ежеквартально, фиксированные даты — вычисляется

Для вычисляемых пунктов — спросить один раз исходные данные (налоговый режим, дата регистрации ИП, есть ли сотрудники) прямо в этой вкладке (не в тесте!), и дальше система сама считает и создаёт нужные дедлайны.

*Журналы*
- Обрабатываются отдельно, см. Этап 4 (генерация + допечатка).

## Этап 3 — Генерация журналов с уникальным QR

Это система, которая сама создаёт готовый к печати PDF конкретной компании — не просто раздача статичных шаблонов.

1. Пользователь в разделе журналов нажимает "Создать журнал" → выбирает тип из списка (УФ-установка, стерилизаторы, предстерилизационная обработка, инструктаж, дезсредства — 5 готовых дизайнов есть, остальные будут присланы позже).
2. Система создаёт запись: уникальный номер журнала + QR-код, привязанные к компании и типу журнала.
3. QR встраивается в PDF на предусмотренное место на обложке (в готовых дизайнах уже зарезервирована зона под QR).
4. Готовый персональный PDF отдаётся пользователю на скачивание. Дальше — печатает сам, где угодно. Никакой логики заказа/оплаты за печать не нужно, это часть подписки.

**Структура файлов-шаблонов:** завести папку в проекте (например `assets/journal-templates/`), по одной подпапке на тип журнала. Для типов, дизайн которых ещё не готов — завести подпапку-заглушку с файлом-заметкой "шаблон появится позже"; сам тип уже присутствует в списке выбора у пользователя, генерация PDF для него временно показывает "скоро будет", но запись о журнале (номер, QR, дедлайны) создаётся и работает полноценно уже сейчас.

**На будущее (не реализовывать в этом этапе, только зарезервировать поле):** булево поле "ведётся ли цифровой дубль" у записи журнала — задел под будущую фичу параллельного цифрового ведения журнала, которую продумаем отдельным пакетом позже.

---

## Этап 4 — Допечатка журнала (окончание страниц)

1. При генерации журнала сохраняется дата создания и количество страниц ежедневного учёта (известно из шаблона).
2. Оценка даты окончания — только грубая, по разумному дефолту для типа журнала (например, "обычно хватает на 6–12 месяцев" — задать вручную для каждого типа, не пытаться точно вычислить, это невозможно без цифрового дублирования).
3. За 2–3 недели до расчётной даты — создаётся карточка-дедлайн "Страницы журнала [название] скоро закончатся" с кнопкой "Сгенерировать новый" (запускает Этап 3 заново для этого типа/компании).

---

## Этап 5 — Главный экран: "Центр действий"

Единый блок на главном экране, объединяющий дедлайны и действия из ВСЕХ источников (тест, "Мои сроки", сотрудники, журналы, расходники, финансы):
- Не группировать по разделу-источнику визуально — только по срочности (просроченное и сегодняшнее — сверху).
- Действия без даты — отдельным блоком сразу после срочных дедлайнов.
- Ограничить видимый список (топ 5–7), с переходом в полный список по клику "Показать все".

---

## Этап 6 — Навигация

Изменить порядок разделов нижней навигации на: **Главная → Безопасность → Финансы → Ещё**.

---

## Порядок реализации (рекомендуемый)

1. Этап 1 (расширение движка дедлайнов) — фундамент.
2. Этап 2 (вкладки в "Безопасность": тест + мои сроки, включая пересмотр PDF-отчёта теста).
3. Этап 3 (генерация журналов с QR) — можно параллельно с этапом 2.
4. Этап 4 (допечатка) — после этапа 3, самый изолированный кусок.
5. Этап 5 (главный экран) — после того как все источники дедлайнов подключены.
6. Этап 6 (навигация) — можно сделать в любой момент, не зависит от остального.

---

## Не в этом этапе (сознательно отложено)

- Цифровое дублирование журналов — отдельный будущий пакет, сейчас только зарезервировать поле (см. Этап 3).
- Дизайн/визуальный стиль приложения и сайта — отдельный этап позже, не блокирует функциональность.
- Партнёрская модель (лиды сторонним компаниям) — не текущий этап.
- Шаблоны журналов, которых ещё нет в дизайне — появятся в проекте по мере готовности, сейчас достаточно структуры-заглушки.
