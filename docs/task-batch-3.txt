## ЭТАП 1 — Флаг модулей по компании + переработка Финансов (ФУНДАМЕНТ)

### 1.1 Флаг модулей
- Добавить в таблицу companies (или отдельную таблицу company_modules)
  список включаемых модулей: visits_clients (Визиты/Клиенты). Начать
  именно с этого модуля — остальные (safety/finance) всегда включены
  для всех.
- Для существующих компаний владельца (его студии) — включить
  visits_clients = true по умолчанию, зафиксировать явно в миграции.
- Для новых компаний — по умолчанию false.
- На фронтенде — если модуль выключен, разделы "Визиты" и "Клиенты"
  не показываются в навигации, роуты возвращают 403 при прямом
  обращении.
- ВАЖНО: отключение модуля НЕ удаляет существующие данные и связи.
  Только скрывает интерфейс.

### 1.2 Переработка Финансов — двойной источник
- В финансовой записи добавить поле source: 'auto_from_visit' или
  'manual'.
- Если модуль visits_clients включён — при внесении визита
  автоматически создаётся finance-запись с source='auto_from_visit',
  привязанная к visit_id.
- Владелец может добавить 'manual' запись в любой момент, независимо
  от наличия visits_clients. Обе записи суммируются, не конфликтуют.
- На фронтенде каждая запись помечена бейджем: "Авто · Визит №123"
  или "Вручную".
- Поле "сотрудник" (membership_id) в финансовой записи —
  ОПЦИОНАЛЬНОЕ (nullable). Аналитика должна корректно работать и с
  разбивкой по сотрудникам, и без неё (просто общие суммы).
- Ввод — без фиксированной периодичности, доступен в любой момент.
- Сравнение периодов (уже частично есть) — убедиться, что работает с
  произвольным диапазоном дат, не только пресетами.

### Чек-лист проверки Этапа 1
- [ ] У студий владельца модуль Визиты/Клиенты виден и работает как раньше
- [ ] У новой тестовой компании модуль Визиты/Клиенты скрыт полностью
- [ ] Финансовая запись от визита помечена "Авто", видна привязка к визиту
- [ ] Можно добавить финансовую запись вручную без привязки к визиту
- [ ] Финансовая запись без указания сотрудника корректно отображается в отчётах
- [ ] Сравнение произвольного периода (не пресета) работает верно

## ЭТАП 2 — Единый движок сроков и уведомлений

- Создать общую таблицу deadlines/notifications с полями: company_id,
  category ('legal'|'tax'|'financial'|'staff'), title, due_date,
  status, related_entity_type, related_entity_id, created_at.
- Любой модуль может регистрировать сюда события со сроком, не
  создавая свою отдельную систему напоминаний.
- Раздел в интерфейсе: "Дедлайны" (замена/сужение прежней идеи
  полноценного календаря) — список предстоящих сроков, отсортирован
  по дате, с фильтром по категории.
- В Настройках — раздел "Уведомления" с тумблерами по категориям
  (юридические/налоговые/финансовые/кадровые) — включить/выключить.
- Пока НЕ делать полноценный календарь с созданием произвольных
  событий и напоминаниями с доставкой — это заморожено. Делать именно
  список дедлайнов.
- Существующий недоработанный календарь (с багом белого экрана при
  смене месяца, багом создания события) — заменить этим разделом, а
  не чинить старую реализацию.

## ЭТАП 4 — Налоговые напоминания

- В настройках компании — поле "налоговый режим" (патент/УСН/др. —
  сделать простой список, расширяемый).
- В зависимости от режима — генерировать напоминания в движке
  уведомлений (категория 'tax') по стандартным срокам (например,
  патент — оплата до определённой даты, страховые взносы — до конца
  года). Реализовать для наиболее распространённых режимов, отметить
  явно в коде, что список сроков может потребовать проверки
  бухгалтером/юристом перед реальным использованием.
- Кто видит: Владелец — всегда. Администратор — решить по аналогии с
  доступом к прибыли (сейчас Администратор не видит прибыль) —
  ПРЕДЛАГАЮ: тоже скрыть от Администратора, пометь как решение по
  умолчанию, которое можно изменить.

### Чек-лист проверки Этапа 4

## ЭТАП 5 — Журналы УФ-лампы и инструктажа на рабочем месте

- Хранить структуру журналов в БД (таблица, редактируемая через
  админку — как юридические документы), НЕ вшивать в код.
- Журнал УФ-бактерицидной установки: простая форма — включил/
  выключил, ответственный (свободный выбор из сотрудников компании,
  не привязан жёстко к роли), время.
- Журнал инструктажа на рабочем месте: разовое событие — кто провёл
  + кто получил, оба подтверждают действие с меткой времени (без
  электронной подписи, просто "подтверждаю").
- Обязательный дисклеймер под каждым журналом: "Ведётся в
  информационных целях и не заменяет обязательную печатную версию,
  предусмотренную законодательством".
- Возможность экспорта журнала в печатаемый PDF.

### Чек-лист проверки Этапа 5
