# Масштаб и ИИ — расширение видения

## Реальный масштаб цели

Studio OS / Безопасный бизнес — это не продукт для одной ниши. Конечная цель —
платформа, которая со временем встраивается в повседневную работу огромного
количества компаний малого и среднего бизнеса, в разных отраслях и разных
странах. Ставка делается на построение очень крупной компании (потенциально
$100 млрд+ по капитализации). Это означает, что решения об архитектуре нужно
принимать так, будто через 5-10 лет платформой будут пользоваться сотни тысяч
компаний одновременно, а не десятки.

Практическое следствие: там, где выбор стоит между "сделать быстрее, но
захардкодить под маникюр" и "сделать чуть дольше, но обобщённо" — выбирать
обобщённый вариант, если разница в трудозатратах разумная. Но не переусложнять
там, где обобщение не нужно прямо сейчас (см. принцип MVP в vision.md).

## ИИ как инфраструктурный слой, а не модуль

ИИ не должен быть "ещё одной кнопкой в меню". Он должен со временем анализировать
данные всей платформы, находить проблемы в бизнесе пользователей, давать
рекомендации, предупреждать о рисках. Чтобы это стало возможным в будущем без
переписывания всего бэкенда, нужно уже сейчас закладывать техническую основу:

1. Единый событийный лог (audit/event log): каждое значимое действие в системе
   (создан визит, списан расходник, отмечен чек-лист, устранено нарушение
   безопасности и т.д.) должно логироваться в едином, структурированном формате
   — это будущее топливо для ИИ-аналитики. Не обязательно строить полноценный
   event sourcing прямо в MVP, но структура данных должна позволять это
   добавить позже без разрушения существующих таблиц.
2. Данные разных модулей должны быть связаны через понятные внешние ключи на
   company_id и user_id, чтобы в будущем ИИ мог сопоставлять данные из разных
   модулей одной компании (например, финансы + чек-листы + инциденты
   безопасности) для комплексных выводов.
3. API должен быть спроектирован так, чтобы в будущем к нему можно было
   подключить отдельный ИИ-сервис (как отдельный микросервис или модуль),
   читающий данные через тот же API или напрямую из базы с правами только на
   чтение — не заводить архитектуру, которая жёстко привязывает бизнес-логику
   к одному монолитному процессу без возможности расширения.

Сейчас НЕ нужно реализовывать саму ИИ-аналитику — только не создавать
архитектурных барьеров, которые сделают её добавление болезненным позже.

## Процесс постоянного притока новых идей

Со временем в проект будет добавляться много новых идей, требований и целых
новых модулей — это нормальный процесс роста, а не разовое ТЗ. Чтобы это не
превращалось в хаос:

1. Все новые идеи/ТЗ складываются в папку docs/ideas/ по мере поступления,
   каждая идея — отдельный файл с понятным названием.
2. Claude Code, встретив новую идею в этой папке, должен сначала оценить её
   на соответствие текущей архитектуре (впишется ли как отдельный модуль без
   переписывания существующего кода), и только после этого предлагать план
   реализации — не бросаться писать код по каждой идее сразу.
3. Любой новый модуль по умолчанию проектируется так, чтобы его можно было
   включать/отключать для конкретной компании (не все компании должны получать
   все модули сразу — см. принцип модульности в vision.md).
