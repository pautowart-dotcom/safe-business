# Безопасный бизнес — новая стратегия и архитектура
*Собрано по итогам стратегического обсуждения · 20 июля 2026*

---

## ЧАСТЬ 1. НОВОЕ ПОЗИЦИОНИРОВАНИЕ

**Было:** операционная система для студий красоты (конкурирует с Yclients/DIKIDI по количеству функций).

**Стало:** платформа защиты владельца малого бизнеса от риска — юридического, санитарного, финансового, кадрового. Заменяет функции, которые малый бизнес не может себе позволить нанять:

- **Раздел "Безопасность" = внутренний юрист** (документы, соответствие регламентам, шаблоны, в будущем — конструктор документов и настоящий/ИИ-юрист за доп. плату)
- **Раздел "Финансы" = внутренний бухгалтер** (аналитика, в будущем — ИИ-консультант)
- **Визиты/Клиенты** — операционный модуль для собственных студий владельца, опциональный (включаемый) модуль для внешних продаж

Название "Безопасный бизнес" теперь буквально описывает весь продукт, а не один из модулей.

**Порядок разделов в интерфейсе:** Главная → Безопасность → Финансы → (Визиты/Клиенты, если включено) → остальное.

## ЧАСТЬ 2. АРХИТЕКТУРА МОДУЛЬНОСТИ (ключевое техническое решение)

### Принцип
Один код, одна база данных для всех компаний. Видимость модулей управляется флагом на уровне компании (`company.enabled_modules` или аналог), а не форком/копией проекта. Отключение модуля скрывает интерфейс и маршруты, но **никогда не удаляет и не разрывает данные**.

### Visits/Clients (Визиты/Клиенты)
- **Ваши студии:** модуль включён навсегда (флаг закреплён за вашими company_id)
- **Новые внешние клиенты:** по умолчанию выключен, включается по договорённости или как платный модуль в будущем
- **Текущая версия в разработке:** оставляем включённым везде, как сейчас. Переключение на выключение — протестировать вручную позже на одной тестовой компании, посмотреть, как ведёт себя интерфейс

### Финансы — двойной источник данных (важнейшая развилка)
Финансовая запись имеет поле `source`: `auto_from_visit` или `manual`.
- Если модуль Визитов включён — система создаёт `auto_from_visit` записи автоматически при внесении визита
- Владелец может добавить `manual` запись в любой момент (если знает, что доход был, а визит не занесён, или если модуля Визитов вообще нет)
- Оба типа записей визуально помечены бейджем ("Авто · Визит №123" / "Вручную"), суммируются вместе, не конфликтуют
- Поле "сотрудник" в финансовой записи — **необязательное**, работает и с разбивкой по мастерам, и без неё
- Ввод — в любой момент по желанию владельца, никакой фиксированной периодичности (не раз в неделю, не ежедневно — когда удобно)
- Сравнение периодов — произвольный диапазон дат, не только пресеты

## ЧАСТЬ 3. ЖУРНАЛЫ (расширение раздела "Безопасность")

### Принцип: не дублировать ввод, использовать уже существующие данные там, где можно

| Журнал | Как заполняется | Источник данных |
|---|---|---|
| Журнал получения/расходования дезсредств | Автоматически | Модуль "Расходники", позиции с тегом "дезинфицирующее средство" |
| Журнал дезинфекции/уборки | Полуавтоматически | Встроен в чек-лист открытия/закрытия смены (доп. пункты) |
| Журнал УФ-бактерицидной установки | Вручную | Простая форма: включил/выключил, ответственный, время. Поле "ответственный" свободное (не привязано жёстко к роли — дежурят разные люди) |
| Журнал инструктажа на рабочем месте | Вручную | Разовое событие: кто провёл + кто получил, оба подтверждают (без электронной подписи — метка времени + "подтверждаю") |
| Журналы стерилизации | Вручную (по присланным вами образцам) | Ожидает ваших примеров для проработки — место в архитектуре заложено заранее |

### Технические требования
- Хранить структуру журналов в БД (таблица, редактируемая через админку), как юридические документы — **не вшивать в код**, чтобы можно было менять содержание без деплоя
- К каждому журналу — понятная встроенная инструкция по заполнению (то, чего нет у покупных бумажных бланков — это ваше конкурентное преимущество)
- Обязательный дисклеймер под каждым журналом: *"Ведётся в информационных целях и не заменяет обязательную печатную версию, предусмотренную законодательством"*
- Печатные PDF-версии журналов — да, как экспорт из софта. Физическая печать и продажа готовых бумажных журналов — отдельное направление, отложить до появления реального спроса (это уже не софт, а логистика/типография)

## ЧАСТЬ 4. ЕДИНЫЙ ДВИЖОК СРОКОВ И УВЕДОМЛЕНИЙ

Вместо трёх отдельных систем напоминаний (безопасность / налоги / финансы) — один общий механизм, куда каждый модуль просто регистрирует свои события со сроком и категорией.

**Категории:** юридический, налоговый, финансовый, кадровый.

**Примеры событий, которые сюда стекаются:**
- Истечение медицинской книжки/сертификата сотрудника
- Налоговые сроки (патент, страховые взносы — по налоговому режиму компании)
- Заканчивающийся расходник
- Финансовая аномалия (выручка заметно ниже обычного)
- Просроченный чек-лист/журнал

**Календарь как таковой (полноценный планировщик с произвольными событиями) — замораживается.** Вместо него — узкая, честная версия: **список дедлайнов и напоминаний**, куда стекаются события из движка выше. Это не отменяет саму идею календаря полностью, а сильно сужает scope до реализуемого и осмысленного.

**Push-уведомления:**
- Технически реализуемо через Web Push API + Service Worker
- **Важное ограничение iOS:** работает только если пользователь добавил сайт на "Домашний экран" через Safari — из обычной открытой вкладки push не придёт. Нужна подсказка в UX об этом при первом входе
- Раздел "Уведомления" в Настройках — тумблеры по категориям (юридические/налоговые/финансовые/кадровые), включать/выключать по отдельности

## ЧАСТЬ 5. РОЛИ И ЛИЧНЫЕ КАБИНЕТЫ — КАК ДОЛЖНО РАБОТАТЬ

*(Требует финального прохода и подтверждения перед реализацией — черновая логика ниже)*

| | Владелец | Администратор | Мастер | Управляющий (будущее) |
|---|---|---|---|---|
| Безопасность (журналы, документы) | Полный доступ, редактирование шаблонов | Просмотр + заполнение | Заполнение своих (смена, УФ, инструктаж) | Вероятно как Администратор |
| Финансы | Полный доступ, включая прибыль | Выручка/расходы, без итоговой прибыли | Нет доступа | Вероятно как Администратор |
| Визиты/Клиенты (если включено) | Полный доступ | Полный доступ | Свои визиты | Полный доступ |
| Уведомления/сроки | Все категории | Все, кроме, возможно, налоговых (обсудить) | Только свои (например, своя мед. книжка) | Как Администратор |
| Экспорт данных | Да | Нет (уже решено ранее) | Нет | Обсудить |

**Открытый вопрос для решения перед реализацией:** должен ли Администратор видеть налоговые напоминания (это финансово чувствительная информация, близкая к прибыли) — решить по аналогии с уже принятым решением "Администратор не видит прибыль".

## ЧАСТЬ 6. НАКОПЛЕННЫЙ СПИСОК ПРАВОК (перенесено из предыдущей сессии тестирования)

### Функциональные баги
1. Онбординг — кнопка на последнем шаге зависает, инструкция не закрывается без F5
2. Календарь — белый экран при смене месяца *(вероятно неактуально после заморозки полноценного календаря — заменить на движок дедлайнов из Части 4)*
3. Календарь — событие не создаётся *(аналогично, вероятно снимается новой архитектурой)*
4. Склад расходников — пропали категории/разделы (было минимум два: работа/бар)
5. Замедление загрузки сайта после багов 1-2 (проверить отдельно)

### UX/дизайн главного экрана
6. Дублирование индекса безопасности (показан дважды)
7. Формулировка "Внимание вчера" — переформулировать, звучит тревожно
8. Разнобой цветов текста между блоками — привести к единому стилю
9. "Добрый день" → заменить на название организации или уменьшить визуальный вес
10. Блок "Задачи на сегодня" — пересмотреть ценность в свете нового позиционирования (вероятно, сливается с движком дедлайнов из Части 4)

### Мелкое
11. Кнопка "Поделиться" (иконка — квадрат со стрелкой вверх) рядом с кнопкой копирования пригласительной ссылки
12. **Иконка при добавлении на "Домашний экран" (iOS/Android)** — сейчас показывается заглушка (первая буква названия). Нужно добавить `apple-touch-icon` + иконки в `manifest.json`, используя логотип компании (готовый файл лого предоставлен владельцем — серый фон, текст "Безопасный бизнес" с диагональными линиями). Технический нюанс: сама иконка домашнего экрана — статичный файл, кэшируется устройством после добавления; "быстрая замена" означает замену файла в проекте при деплое, а не смену без деплоя через админку (это ограничение самой технологии на всех платформах, не наше решение)

## ЧАСТЬ 7. ПРЕДЛАГАЕМЫЙ ПОРЯДОК РЕАЛИЗАЦИИ

1. **Фундамент:** флаг модулей по компании + переработка схемы Финансов (двойной источник, необязательный сотрудник, произвольный период)
2. **Единый движок сроков/уведомлений** (техническая основа для всего остального)
3. **Мед. книжки/сертификаты сотрудников с напоминаниями** (быстрая, ощутимая ценность)
4. **Налоговые напоминания** (по налоговому режиму компании)
5. **Журналы УФ-лампы и инструктажа** (простой ручной ввод)
6. **Автоматические журналы** (дезсредства из Расходников, дезинфекция из чек-листов смены)
7. **Журналы стерилизации** — после получения ваших образцов
8. **Кнопка "Сформировать досье"** (PDF-выгрузка истории по клиенту/дате/мастеру)
9. **Push-уведомления + раздел настроек уведомлений**
10. Закрытие накопленного списка багов/UX-правок (Часть 6) — вплетать по ходу, где логически пересекается с новой архитектурой
