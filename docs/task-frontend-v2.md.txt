# Задача: привести фронтенд к дизайну прототипа + починить вход

## 1. Баг входа (приоритет — сначала это)
После логина Super Admin видит "У вас нет доступа ни к одной компании".
Нет способа создать первую компанию через интерфейс.
Нужно: экран/форма создания компании для Super Admin (или автоматическое
создание компании при первом входе, если компаний ноль).

## 2. Дизайн — ориентир reference/studio_os_mvp.tsx

Взять из прототипа:
- Мобильный формат, максимум 430px в ширину, даже в браузере на десктопе
- Нижняя навигация (bottom nav) вместо бокового меню
- Липкий верхний хедер с названием раздела и аватаркой владельца справа
- Компоненты: Card (белая карточка, рамка, радиус 14px), ST (мелкий
  серый заголовок заглавными), Btn, Badge, инлайн SVG-иконки
- Разделение по ролям:
  - Владелец: Главная / Клиенты / Визиты / Финансы / Ещё
  - Мастер: Главная / Клиенты / Смена / Склад / Финансы / Ещё
    (у мастера видны только свои клиенты и своя зарплата, не вся студия)
  - Внутри "Ещё" у владельца: Расходники, Смена, База знаний,
    Безопасность, Обратная связь, Команда, Настройки
  - Внутри "Ещё" у мастера: только База знаний

## 3. Цветовая схема — НЕ строго чёрно-белая

Взять графитовый вместо чистого чёрного как основной акцентный цвет
(например #2A2A2E) для кнопок и активных элементов.
Оставить как в прототипе: зелёный/оранжевый/красный для статусов.
Позже добавим фирменные цвета — пока просто не скучно-монохромно.

## 4. Раздел "Безопасность" — важное отличие от прототипа

В прототипе Безопасность — это просто загрузка готового PDF от
специалиста вручную. ЭТО УСТАРЕЛО. У нас уже полностью готов свой
движок (backend/src/modules/security): бесплатный тест на 12 вопросов,
платный аудит на 34 вопроса, автоматический скоринг, матрица нарушений,
генерация PDF-отчёта самой системой.

Взять из прототипа только: визуальный стиль карточки индекса
безопасности, структуру "Ещё → Безопасность" как раздел внутри
приложения (не отдельный продукт), карту нарушений с цветовыми
индикаторами риска.
НЕ переносить логику ручной загрузки PDF — вместо неё подключить уже
готовый backend-движок теста/аудита.

## Общее
Использовать прототип как эталон дизайна и структуры компонентов,
подключая к реальному backend API вместо моковых данных внутри файла.
