## Важно понимать
Это не разовый проект под одну студию маникюра. Это фундамент платформы, которая
должна масштабироваться на тысячи компаний, разные отрасли (маникюр, ресницы,
фитнес, медицина и т.д.) и десятки будущих модулей — без переписывания
архитектуры. Поэтому сейчас архитектурные решения важнее скорости написания кода.

## Требования к архитектуре

1. Мультитенантность с самого начала:
   - Таблица companies (не "studios" — обобщённое понятие, не привязанное к отрасли)
   - Один пользователь может иметь доступ к нескольким компаниям через свой аккаунт
   - Роль Super Admin (я, Артём) — видит все компании всех клиентов платформы
   - Роль Owner — видит все данные своей компании (может быть несколько филиалов)
   - Роль Master/сотрудник — ограниченный доступ по правилам ниже
   - Полная изоляция данных между компаниями обязательна на уровне запросов
     к базе данных, а не только скрыта на фронтенде

2. Модульная структура кода: каждый функциональный блок (визиты, финансы,
   склад, чек-листы, база знаний, безопасность, будущие модули) — отдельный
   независимый модуль, который можно подключать/отключать, не трогая остальные

3. API-first: backend отдаёт чистый REST API, не завязанный на конкретный
   frontend — в будущем на этом же backend должны работать и мобильные приложения

4. Схема базы данных не должна жёстко привязываться к "студии маникюра" —
   нужна обобщённая модель компании/филиала с настраиваемым набором модулей

5. Единый событийный лог (audit/event log): каждое значимое действие в системе
   должно логироваться в структурированном формате — это основа для будущей
   ИИ-аналитики. Не строить полноценный event sourcing сейчас, но не мешать
   добавить это позже.

6. API должен быть спроектирован так, чтобы в будущем к нему можно было
   подключить отдельный ИИ-сервис, читающий данные через API или из базы
   с правами только на чтение.

## Обязательные бизнес-правила (детали в docs/product-context.md)

- Форма визита: фото до/после раздельно, быстрый ввод (Enter между полями),
  поиск клиента по фамилии БЕЗ показа телефона мастеру, скидка в % с
  автопересчётом, зарплата мастера = индивидуальный % от ПОЛНОЙ суммы без скидки
- Телефон клиента НИКОГДА не должен быть виден мастеру — это должно быть
  реализовано на уровне API (сервер не отправляет это поле в ответе для роли
  мастера), а не просто скрыто в интерфейсе
- Финансы владельца: авторасчёт зарплат, отдельно фиксированные/процентные/
  переменные расходы, формула чистой прибыли (см. product-context.md)
- Чек-листы редактирует только владелец, мастер только отмечает выполнение
- База знаний: статьи создаёт только владелец, мастер только читает
- Команда: приглашение мастеров по ссылке, удаление — только владельцем
- Обратная связь от мастера передаётся владельцу
- Расходники: ссылка на товар, кнопки "Пришло"/"Списать", индикатор низкого остатка

## Модуль "Безопасный бизнес" (по ТЗ в docs/security-engine/)

- Встроен как раздел внутри платформы, не отдельный сайт
- Бесплатный тест из 12 вопросов → результат/индекс безопасности на странице
- Платный полный аудит → персональная страница с картой нарушений и дорожной
  картой устранения
- Возможность скачать PDF-отчёт
- Отметка "Устранено" по каждому нарушению
- Хранение документов компании по категориям

## Процесс новых идей в будущем

Со временем будет добавляться много новых идей и модулей. Для этого:
1. Новые идеи складываются в docs/ideas/, каждая — отдельный файл
2. Claude Code сначала оценивает идею на соответствие архитектуре, потом
   предлагает план — не пишет код сразу по каждой новой идее
3. Новый модуль по умолчанию можно включать/отключать для конкретной компании
